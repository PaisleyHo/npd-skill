# Merge Rules

这份文件定义：

当你后续追加新的聊天记录、口述、截图、照片信息时，应该怎么把增量内容 merge 进已有 skill。

目标有两个：

- 越补越像
- 不被零碎噪音带偏

## 总原则

1. 新信息不自动推翻旧信息，先判断是否冲突。
2. 高频、稳定、跨场景出现的模式，优先级高于一次性片段。
3. 原话优先级高于抽象评价。
4. 具体事件优先写进 `memory.zh-CN.md`。
5. 稳定表达习惯和防御逻辑优先写进 `persona.zh-CN.md`。
6. 用户明确说“这不像 ta”的纠正，优先写进 `corrections.zh-CN.md`。

## 写入去向规则

### 写入 Persona 的内容

适合写进 `persona.zh-CN.md` 的是：

- 长期稳定的说话节奏
- 常见防御方式
- 被夸、被批评、被忽略时的惯常反应
- 对地位、面子、认可的固定敏感点
- 关系里反复出现的气质和控制方式

判断标准：

- 是否跨多个对话场景重复出现
- 是否不依赖某一个具体事件也成立

### 写入 Memory 的内容

适合写进 `memory.zh-CN.md` 的是：

- 共同经历
- 固定地点
- 纪念日
- inside jokes
- 某次吵架、和好、冷战的具体细节
- 对方说过的代表性原话

判断标准：

- 是否带有具体时间、地点、人物、事件
- 是否一旦删掉，角色就“不像这个人”

### 写入 Corrections 的内容

适合写进 `corrections.zh-CN.md` 的是：

- “ta 不会说这种词”
- “ta 不会这么快低头”
- “ta 生气时不是凶，是特别冷”
- “ta 不会发这么长一段”
- “ta 平时更喜欢阴阳，不会正面冲”

判断标准：

- 是否直接针对之前某次错误输出
- 是否属于高优先级修正

## 冲突处理

当新旧信息冲突时，按这个顺序判断：

1. 用户明确纠正
2. 原始聊天记录中的高频模式
3. 多次重复出现的行为
4. 单次事件
5. 用户模糊印象

如果存在“两种状态都是真的”，不要硬删其中一个，应改成：

- 在 `persona.zh-CN.md` 里写成条件触发
- 在 `memory.zh-CN.md` 里补充对应场景

例如：

- 平时很会讲话
- 但受辱后会突然非常短、非常冷

这不是冲突，而是状态切换。

## 增量更新流程

每次追加材料时，建议按这个顺序操作：

1. 先提取新材料中的原话、事件、冲突点、口头禅
2. 标记哪些是“稳定模式”，哪些是“单次事件”
3. 把稳定模式写进 `persona.zh-CN.md`
4. 把具体事件和原话写进 `memory.zh-CN.md`
5. 如果发现以前写错了，写入 `corrections.zh-CN.md`
6. 在 `version-history.zh-CN.md` 里登记这次更新

## 防漂移规则

为了避免 skill 越更越偏，合并时注意：

- 不要因为一两句温柔就把整个人设写软
- 不要因为一次激烈争吵就把整个人设写成纯攻击型
- 不要把用户的情绪投射直接写成人设事实
- 不要把抽象判断替代原话和事件
